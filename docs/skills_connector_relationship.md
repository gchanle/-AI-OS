# Skills / 连接器 / 萤火虫 关系说明

更新日期：2026-03-30

## 结论

`连接器中心` 不应直接等于 `Skills 管理中心`，但两者必须强关联。

在产品入口上，两者更适合被统一收进一个更上层的：

- `能力接入中心`

更合理的三层关系是：

- `连接器（Connector）`：系统接入层
- `Skill`：能力封装层
- `萤火虫（Firefly）`：任务理解与调用编排层

## 1. 连接器解决什么问题

连接器主要回答：

- 怎么进入某个系统
- 用什么授权方式进入
- 是否需要凭证保险库
- 怎么巡检与调试
- 调用后返回什么结构化回执

它的边界是“系统接入能力”，不是“用户最终看到的能力单元”。

例如：

- 教务系统连接器
- 通知中心连接器
- 图书馆馆藏连接器
- 波尔科研平台连接器

## 2. Skill 解决什么问题

Skill 主要回答：

- 哪个能力应该给谁用
- 这个能力属于 `AI 办事 / AI 科研 / AI 助教 / AI 图书馆 / AI 智能体` 哪一块
- 它依赖哪些连接器
- 它是不是学校官方、超星官方或个人自建
- 它能不能进入学校的 Skills 市场
- 它是否允许被萤火虫调用

它的边界是“能力产品化与治理”。

例如：

- `课表摘要助手`
- `通知分拣员`
- `AI 阅读伴读`
- `科研线索雷达`
- `智能体上架顾问`

## 3. 萤火虫解决什么问题

萤火虫主要回答：

- 用户现在要完成什么事
- 应该调哪个 Skill 更合适
- 这个 Skill 背后是否还要再调连接器
- 返回给用户时，如何把系统结果整理成自然的任务推进与对话结果

它的边界是“入口与编排”，不是“技能治理”或“系统接入配置”。

## 4. 为什么不能把 Skills 直接塞进连接器中心

如果把 Skills 直接做成连接器的一部分，会出现四个问题：

- 系统接入配置和能力上架审核混在一起
- 学校官方 Skill、超星 Skill、个人 Skill 没有统一治理视图
- 无法表达“一个 Skill 绑定多个连接器”的关系
- 萤火虫最终调用的是 Skill 还是 Connector 会变得模糊

所以更合理的做法是：

- 连接器中心：看“系统接入是否可用”
- Skills 管理中心：看“能力是否可以被启用和上架”
- 萤火虫：看“当前任务要不要调用这个能力”

## 5. 适合本项目的落地方式

建议结构如下：

- `能力接入中心`
  - 连接器
  - Skills
  - MCP
  - CLI / 客户侧代理
  - 凭证保险库
- `连接器中心`
  - 系统台账
  - 授权与凭证
  - 巡检 / 调试 / 审计
- `Skills 管理中心`
  - 官方 Skills
  - 学校 Skills 市场
  - 个人自建 Skills
  - 提交审核 / 上架 / 下架
  - 是否启用给萤火虫
- `萤火虫`
  - 根据用户意图调用 Skills
  - Skill 再决定是否调连接器

## 6. 对产品语义的直接好处

这样处理后，`AI 办事` 就不再只是一个页面入口，而是可以明确表达为：

- `AI 办事` = 一组被启用的办事 Skills

同理：

- `AI 图书馆` = 阅读、伴读、笔记、馆藏联动等 Skills 的组合
- `AI 科研` = 检索、归纳、平台接力等 Skills 的组合
- `AI 智能体` = 智能体创建、审核、上架与分发相关 Skills 的组合

这比“每个一级栏目下面只是挂几个外部系统入口”要更像真正的 `AI 校园 OS`。
