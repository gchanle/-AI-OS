# AI 图书馆产品蓝图

Updated: 2026-03-26

## 产品定位

`AI 图书馆` 不应该只是一个资料检索页，而应该是面向高校用户的阅读工作台。

目标体验更接近：

- 一部分像微信读书
- 一部分像高校线上图书馆
- 一部分像与 `萤火虫` 联动的阅读助手

它要解决的不只是“找到书”，而是：

- 找书
- 选书
- 放进书架
- 持续阅读
- 记录进度
- 生成笔记
- 在阅读过程中随时和 AI 对话
- 回连学校真实馆藏与借阅系统

## 核心功能结构

### 1. 图书广场

这是发现入口，负责：

- 分类浏览图书、名著、历史、地理、科研、专业教材等
- 展示推荐书单、专题书单、热门借阅、馆藏精选
- 允许用户搜索书名、作者、主题词
- 把感兴趣的书加入书架

理想感受：

- 像一个可逛的线上图书馆
- 有“广场感”，而不只是结果列表

### 2. 我的书架

书架是用户的持续阅读空间，负责：

- 存放从广场加入的书
- 展示最近阅读
- 保留阅读进度
- 支持继续阅读
- 支持从书架移除

这里是 `AI 图书馆` 的高频入口之一。

### 3. 阅读器

任意一本书点开后，都应该进入专门的阅读页面。

阅读器至少应包含：

- 接近微信读书的沉浸式阅读区
- 左右翻页或章节切换
- 当前进度显示
- 高亮、摘录、批注
- 加入书架

阅读器不是资料详情页，而是可持续阅读的主界面。

### 4. 阅读中的 AI 对话

阅读器右侧应保留对话区域，由 `萤火虫` 提供上下文感知能力。

核心要求：

- AI 知道当前打开的是哪本书
- AI 理解当前章节、当前段落或当前引用内容
- 用户可以随时提问、解释、总结、延伸
- 对话结果可转成笔记、摘要或任务

这部分是 `AI 图书馆` 与普通电子书产品最大的差异点。

### 5. 阅读统计

需要形成可持续的阅读反馈，包括：

- 年度阅读本数
- 累计阅读本数
- 阅读时长
- 本月阅读进度
- 最近阅读连续天数

这部分能增强长期使用动力，也方便学校后续做学习空间运营。

### 6. 笔记系统

笔记应是独立能力，不只是一个大文本框。

至少应支持：

- 一条条笔记记录
- 关联书名、章节、引用段落
- 记录创建时间与更新时间
- 列表查看
- 详情查看与编辑
- 富文本基础能力

推荐最小字段：

```ts
type LibraryNote = {
  id: string;
  bookId: string;
  bookTitle: string;
  quote?: string;
  chapter?: string;
  content: string;
  createdAt: string;
  updatedAt: string;
}
```

### 7. 学校图书馆联动

这是 B 端价值的关键来源。

最终希望支持：

- 查询学校馆藏
- 查询电子书与期刊可用性
- 查询我的借阅状态
- 查看哪些书已归还、哪些即将到期
- 连接学校已有图书馆系统

这部分应优先走：

1. 官方接口 / 数据服务
2. 单点登录与深链接
3. 必要时再考虑只读型浏览器自动化

## 建议的信息架构

建议后续把 `AI 图书馆` 拆成这些一级结构：

1. 图书广场
2. 我的书架
3. 阅读器
4. 我的笔记
5. 阅读统计
6. 馆藏联动

其中：

- `阅读器` 是核心工作面
- `我的书架` 是用户沉淀面
- `我的笔记` 是知识沉淀面
- `馆藏联动` 是高校差异化能力

## 当前页面的主要差距

当前版本更像“资料检索与 Firefly 交接页”，还不是完整可用的 `AI 图书馆`。

主要差距有：

- 还没有真正的书架机制
- 还没有独立阅读器
- 还没有上下文感知的阅读对话
- 还没有阅读统计面板
- 还没有结构化笔记系统
- 还没有与学校馆藏做真实联动

## 推荐实现顺序

### Phase 1

- 图书广场
- 加入书架
- 我的书架
- 阅读器基础版
- 阅读器右侧 AI 对话

### Phase 2

- 结构化笔记
- 引用摘录
- 阅读进度与阅读统计

### Phase 3

- 学校馆藏联动
- 借阅状态
- 电子书 / 期刊 / 馆藏统一检索

## 与萤火虫的关系

`AI 图书馆` 不应只是跳转给 `萤火虫`，而应在馆内就有 AI。

更合理的关系是：

- 图书馆负责“阅读场景”
- 萤火虫负责“任务推进、解释、整理、跨系统联动”

两者之间应该互相回流，而不是单向跳转。
