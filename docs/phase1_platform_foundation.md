# Phase 1 平台中台层收口说明

更新日期：2026-03-30

## 目标

这一阶段不追求新增更多页面，而是先把平台级公共语义收口，避免以下问题继续扩大：

- 同一个平台概念在多个文件里重复定义
- `localStorage` key 到处散落
- 萤火虫 handoff 参数在不同模块里各自拼接
- 消息来源、模块来源、业务入口不是同一套注册表
- 页面能跑，但后续很难交给研发做真实分层

## 这一阶段优先收口的内容

### 1. 平台模块注册表

统一维护以下模块元信息：

- `萤火虫`
- `AI 办事`
- `AI 科研`
- `AI 助教`
- `AI 图书馆`
- `AI 智能体`
- `连接器中心`
- `系统通知`

每个模块至少统一这些字段：

- `id`
- `label`
- `href`
- `capabilityId`
- `source`
- `summary`
- `kind`

核心文件：

- `data/campusPlatform.js`

### 2. 工作台偏好存储

将共享工作台偏好收口到统一入口，避免页面自行读写：

- `campus_workspace_prefs`
- 模型偏好
- 能力偏好
- 工作台模式
- dashboard section 偏好

### 3. 萤火虫 handoff 统一

统一以下逻辑：

- `firefly_prompt`
- `firefly_caps`
- 构造 handoff URL
- 消费 handoff 请求

这意味着：

- 图书馆、连接器、外部工作区之后都不用各自手写 handoff 参数拼接

### 4. 平台级事件统一

统一以下事件语义：

- `campus-notify`
- `campus-message-sync`
- `campus-open-firefly`

这样消息中心、右侧萤火虫、连接器回执、工作区协同能用同一套平台事件工作。

### 5. 消息来源统一

消息页、消息详情、顶部消息中心、平台模块注册表，后续都应围绕同一套来源映射：

- `messageSourceOrder`
- `messageSourceMap`
- `buildMessageSourceTabs`
- `getCampusMessageSource`

目标是：

- 不再由 `messages page`、`messageCenter`、`detail page` 各自维护一套来源标签

### 6. Skills 能力层补齐

在平台层面还需要补齐一层明确的 `Skills` 语义：

- 连接器负责系统接入
- Skill 负责能力封装、审核与上架
- 萤火虫负责调用编排

这一步的目的不是新增一个“介绍页”，而是把：

- 学校官方 Skills
- 超星官方 Skills
- 个人自建 Skills
- 学校 Skills 市场
- 萤火虫可调用 Skills

放进一套明确的数据结构和管理页面里，避免后续继续把“能力”和“系统接入”混在一起。

### 7. 能力接入入口统一

随着 `Skills / MCP / CLI / 凭证治理` 继续加入，导航层不适合把它们拆成多个并列一级入口。

更合理的方式是：

- 用一个统一的 `能力接入中心` 承接所有“能力进入校园 OS”的方式
- 其中 `连接器`、`Skills`、`MCP`、`CLI`、`凭证保险库` 作为子模块存在
- 避免信息架构越来越碎，影响 B 端后台的一致性

## 本轮已完成

- 新增 `data/campusPlatform.js`
- 首页工作台偏好改用统一读写入口
- 首页 handoff 消费逻辑改用统一入口
- 外部工作区 handoff 改用统一 URL 构造
- AI 图书馆 handoff 改用统一 URL 构造
- 连接器中心消息通知与 handoff 改用统一平台入口
- 右侧萤火虫抽屉共享模型偏好改用统一工作台偏好入口
- 顶部消息中心事件监听改用平台层事件定义
- 消息页来源 tabs 改为走平台层来源构造
- 消息详情页来源说明改为读平台模块元信息
- 新增 `Skills 管理中心` 的基础数据层、页面骨架和文档说明
- 连接器中心开始展示“当前连接器被哪些 Skill 依赖”

## 这一阶段还要继续完成的内容

### 1. 连接器 / 消息中心 / 萤火虫的事件链统一

下一步应继续明确：

- 哪些动作会发消息
- 哪些动作会打开萤火虫
- 哪些动作只更新状态不推送通知

### 2. 平台级数据结构继续抽离

至少要继续抽离这些方向：

- Firefly 会话元信息
- 连接器运行态摘要
- 授权 / 保险库引用关系
- 消息与任务之间的关系

### 3. 文档对齐

后续应把：

- 平台模块边界
- 消息中心边界
- 萤火虫边界
- 连接器中心边界
- Skills 管理中心边界

整理成研发能直接理解的模块拆分文档。

## 这一阶段的价值

这一步的价值不在于“页面变多了”，而在于：

- 平台语义开始统一
- 以后继续改不会每次都牵一堆重复字符串
- 页面层和平台层开始有边界
- 后续能更顺利过渡到真实接口与研发分工
