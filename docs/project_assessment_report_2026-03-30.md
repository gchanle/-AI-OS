# 超星 AI 校园 OS 项目评估报告

更新日期：2026-03-30

## 1. 执行摘要

当前项目已经从“AI 聊天页面”演进为一个具有明确产品方向的 `AI 校园 OS` 原型，核心概念、主界面结构、主要能力分区和若干关键模块已经初步成型。

从整体判断看：

- 产品概念成熟度：`80%+`
- 演示原型成熟度：`70%-75%`
- 前端可视化完成度：`70%+`
- 研发可接手程度：`40%-50%`
- 真实业务接入成熟度：`20%-30%`

一句话判断：

这个项目已经具备继续深做的价值，适合继续在 Codex 上推进；但下一阶段不应继续只靠页面堆叠功能，而应进入“结构收口 + 真实接入设计 + 重点场景闭环”的阶段。

## 2. 项目定位判断

当前产品方向已经比较清晰，不再是一个普通的校园聊天助手，而是：

- 公司品牌：`超星`
- 顶层产品概念：`超星 AI 校园 OS`
- 中心工作台：`萤火虫`
- 产品气质：克制、B 端、系统化、具备“校园操作系统”而非“C 端聊天窗口”的感觉

这个方向是成立的，原因有三点：

- 你希望承载的不只是一个对话入口，而是高校用户跨系统、跨任务、跨场景的统一工作面。
- 目前产品已经有 `AI 办事 / AI 科研 / AI 助教 / AI 图书馆 / AI 智能体` 这样的能力分区，具备 OS 的一级导航雏形。
- `萤火虫` 已经开始承担“统一入口、任务接力、跨模块回流”的角色，而不是孤立对话框。

## 3. 当前完成情况评估

### 3.1 萤火虫主工作台

当前首页已经具备较完整的主工作台框架，核心能力包括：

- `工作台 / 对话` 双模式切换
- 会话恢复与本地偏好记忆
- 模型切换
- 能力切换
- `web search / deep research` 开关
- 外部页面 handoff 回流到萤火虫

核心文件：

- `app/page.js`
- `components/LandingView.js`
- `components/ChatArea.js`
- `data/workspace.js`

评估：

- 这部分已经不是空壳，具备产品骨架。
- 交互方向是对的，尤其是“不是纯聊天”的克制 B 端工作台气质已经开始建立。
- 后续需要做的是状态统一和交互收口，不是推翻重做。

### 3.2 外部系统统一壳层

`AI 办事 / AI 科研 / AI 助教 / AI 智能体` 已经基本建立统一承载方式，主要能力包括：

- 左侧业务导航承载
- 嵌入查看 / 当前窗口 / 新标签 打开模式
- 首次提示与模式记忆
- 加载过渡处理
- 右侧 `萤火虫` 协同抽屉

核心文件：

- `components/ExternalWorkspaceShell.js`
- `components/ExternalOpenModeControl.js`
- `app/services/page.js`
- `app/research/page.js`
- `app/assistant/page.js`
- `app/agent-builder/page.js`

评估：

- 这是当前项目最值钱的抽象层之一。
- 它已经证明“AI 校园 OS”可以统一承接不同业务系统，不需要每个系统都单独重做外壳。
- 但目前仍以“接入承接”能力为主，距离“真实系统能力调用”还差后端与连接器执行层。

### 3.3 AI 图书馆

`AI 图书馆` 是目前最有产品感、最接近独立工作台的模块之一，已经具备：

- 图书广场
- 我的书架
- 阅读器雏形
- 笔记与阅读状态
- 阅读统计方向
- 右侧萤火虫协同

核心文件：

- `app/library/page.js`
- `app/library/library.css`
- `data/library.js`
- `docs/library_product_blueprint.md`

评估：

- 这是全项目最适合继续打磨成标杆模块的一块。
- 它已经开始体现 `AI 校园 OS` 的差异化，不再只是“把外部系统包一层壳”。
- 如果后续要做产品亮点展示，图书馆应被视为优先场景。

### 3.4 AI 科研

`AI 科研` 当前已经形成两个重要判断：

- 面向用户的公开场景仍以 `闻道科学导航` 为主
- `波尔科研空间站` 作为候选平台接入方式被保留

核心文件：

- `app/research/page.js`
- `docs/research_platform_evaluation.md`

评估：

- 方向是谨慎且合理的，没有盲目把不成熟的科研能力硬塞到主路径。
- 目前更像“科研平台承载层”，不是“科研能力中心”。
- 后续应该优先思考科研任务如何真正闭环，而不是继续加更多科研站点入口。

### 3.5 连接器中心 / 凭证保险库 / 授权中心

`连接器中心` 已经开始从介绍页转向真正的管理后台，当前具备：

- 连接器台账页面
- 更强的表格视图
- 新建 / 编辑抽屉表单
- 详情面板
- `凭证保险库` 独立页面
- 与消息中心和萤火虫的关系雏形

核心文件：

- `app/connectors/page.js`
- `app/connectors/[id]/page.js`
- `app/connectors/vault/page.js`
- `components/ConnectorCenter.js`
- `components/ConnectorVaultCenter.js`
- `data/connectors.js`

评估：

- 这是把“只是外链接入”升级为“系统能力管理”的关键一步。
- 产品方向正确，且非常关键，因为它关系到 AI 校园 OS 是否真的像一个平台。
- 当前仍以原型数据和前端状态为主，后续必须补“后端数据结构 + 授权模型 + 审计逻辑”。

### 3.6 模型层与消息中心

当前模型层和消息中心已经具备基础可用性：

- `/api/models` 已支持基于 DashScope 兼容接口动态探测模型
- 已支持多模型候选列表
- 消息中心已支持多来源分类与消息详情页

核心文件：

- `app/api/models/route.js`
- `data/messageCenter.js`
- `components/GlobalMessageCenter.js`
- `app/messages/page.js`

评估：

- 模型层已经达到“可演示、可切换、可扩展”的程度。
- 消息中心已经有产品雏形，但还需要和业务动作、连接器回执、任务完成通知深度打通。

## 4. 当前最主要的问题

### 4.1 架构层已经开始落后于页面层

目前项目最主要的问题不是页面做不出来，而是：

- 页面原型推进速度，已经快于底层结构抽象速度

具体表现：

- 多个模块仍依赖 `localStorage`
- 页面内部状态较重
- 模拟数据与真实数据边界不清晰
- 模块之间存在“各存一套状态”的风险

影响：

- 再继续无序堆功能，后续会越来越难收口
- 研发接手时，拆接口和状态边界会比较痛苦

### 4.2 真实业务接入仍然偏弱

现在多数业务模块更接近：

- 有统一壳层
- 有 OS 感
- 有协同感

但还不等于：

- 已经真正打通业务系统数据
- 萤火虫已经可以稳定调用系统能力

目前较弱的点包括：

- 教务、通知、图书馆馆藏等仍主要停留在原型数据或接入设计层
- 缺少真正的连接器执行器和可追踪调用链
- 缺少统一的授权后端模型

### 4.3 工作区现场较脏，文档需要刷新

当前工作树不是干净状态，且与旧的恢复文档已有偏差。

具体风险：

- 现有改动很多，但最新边界没有被正式整理进统一文档
- `PaperHelper` 已经退出主方向，但删除残留很多
- 如果这时切给研发，容易出现“看页面像一回事，实际边界不清楚”的问题

### 4.4 还没有形成正式的交付分层

如果未来要交给研发，至少应拆出这几层：

- UI 壳层
- 页面编排层
- 连接器中心
- 鉴权 / 保险库中心
- 消息中心
- 萤火虫会话与任务层
- 模型能力层
- 真实业务数据接入层

目前这些层在代码里已经有雏形，但还没有被正式定义清楚。

## 5. 风险评估

### 5.1 产品风险

- 如果继续增加页面和入口，而不优先做重点场景闭环，产品会越来越“像大盘”，但不够“能用”。
- 如果外部系统始终只是 iframe 或导航壳，用户会觉得 OS 很完整，但真正用起来仍需要跳出。

### 5.2 工程风险

- 本地状态分散会让模块联动越来越难维护。
- 页面过重会带来更多重渲染、交互抖动与维护成本。
- 后续若引入真实接口，没有统一的数据契约会导致重构成本上升。

### 5.3 安全与合规风险

- 连接器、保险库、授权中心的方向是对的，但当前还停留在前端原型层。
- 涉及账号、凭证、会话、第三方系统访问时，后续必须明确：
  - 凭证是否落库
  - 谁可见
  - 如何审计
  - 如何失效与回收

## 6. 是否适合继续在 Codex 上推进

结论：`适合，而且建议继续在 Codex 上推进下一阶段。`

原因：

- 当前项目已经形成连续上下文，Codex 这边对现有结构理解更完整。
- 你前面担心的“改得面目全非”问题，已经可以通过分模块、分阶段、小步迭代来控制。
- 连接器中心、图书馆、萤火虫主工作台这些部分，已经开始在 Codex 这边形成连续产品思路。
- 相比 token 受限、上下文易断，当前更需要的是连续迭代和结构收口。

但也要明确：

- Codex 适合继续共创原型、架构、交互与前端演示层
- 真正进入生产交付前，仍需研发团队完成后端与工程体系化实现

## 7. 建议的后续规划

### Phase 0：项目收口与基线冻结

目标：

- 先冻结一级信息架构和核心模块边界

建议完成事项：

- 更新项目现状文档
- 明确哪些模块属于 `OS 原生`
- 明确哪些模块属于 `外部系统承载`
- 明确哪些能力未来走 `连接器调用`
- 清理 `PaperHelper` 相关残留说明

### Phase 1：平台中台层收口

目标：

- 从“页面原型”过渡到“平台结构原型”

建议优先收口：

- 消息中心
- 连接器中心
- 凭证保险库 / 授权中心
- 萤火虫会话与任务模型
- 模型配置层

这一阶段的关键产出不是新页面，而是：

- 统一的数据结构
- 状态边界
- 模块职责

### Phase 2：两条优先场景做深

建议优先做深的两条场景：

#### 场景 A：萤火虫 + AI 办事 + 连接器

目标：

- 让用户看到“萤火虫不是聊天，而是真的能帮我跨系统办事”

建议闭环：

- 通知读取
- 教务查询
- 服务大厅任务引导
- 消息中心回执

#### 场景 B：AI 图书馆 + 阅读器 + 陪读助手

目标：

- 做出真正有记忆、有沉淀、有陪读价值的阅读工作台

建议闭环：

- 图书广场
- 书架
- 阅读器
- 笔记
- 右侧萤火虫上下文协同

### Phase 3：真实接入设计与交付准备

目标：

- 为研发接手做准备

建议产出：

- 连接器执行模型
- 授权与凭证模型
- CAS / SSO / 代理 / 浏览器自动化 接入策略
- 消息与任务事件模型
- 模块级接口草案

## 8. 近期优先级建议

建议后续按以下顺序推进：

1. 先整理模块边界和最新现状文档
2. 再收口连接器中心、保险库、消息中心和萤火虫之间的关系
3. 继续深做 `AI 图书馆`
4. 再推进 `AI 办事 / AI 科研 / AI 助教` 的真实接入策略

不建议继续优先做的方向：

- 继续横向增加大量新页面
- 继续只做视觉修饰而不做结构收口
- 在没有统一状态模型前继续堆复杂功能

## 9. 综合结论

这个项目当前最值得肯定的地方，不是“页面多”，而是：

- 它已经开始有自己清晰的产品语义
- 有 `AI 校园 OS` 的整体感
- `萤火虫`、外部工作区、图书馆、连接器中心之间已经开始形成结构关系

它当前最需要补的，不是“再加一个新模块”，而是：

- 统一结构
- 做深重点场景
- 明确真实接入路径
- 为研发交付预先整理清楚边界

最终判断：

这是一个值得继续投入和继续共创的项目，而且已经过了“要不要做”的阶段，进入了“怎么把它做稳、做深、做成体系”的阶段。
