# 萤火虫验收清单（2026-04-13 晚版）

## 本轮范围

本轮重点不是重做对话页，而是把萤火虫从“单次问答入口”往“任务型 Agent Workbench + 可观测 Runtime”推进一层，主要覆盖：

- 前台 `firefly-workbench`
- 后台 `admin/access?tab=runtime`
- deer-flow 混合 runtime 的线程态、checkpoint、subagent 可视化
- 前台工作台的信息层级和样式细化

## 明天验收入口

1. 前台工作台：`/firefly-workbench`
2. 后台运行观测：`/admin/access?tab=runtime`
3. 可选辅助页：`/assistant`

## 建议验收顺序

1. 先在 `firefly-workbench` 发起一个新任务
2. 再到 `admin/access?tab=runtime` 查看该任务的运行态
3. 找一条失败或可恢复任务，验证 retry / resume
4. 回到前台看同一线程是否同步反映状态

## P0 必须通过

### 1. 前台可以发起任务型会话

- 在 `firefly-workbench` 输入一条任务型问题，点击“启动任务会话”
- 预期：
  - 最近会话出现新会话
  - 任务收件箱数字变化
  - 会话详情出现步骤或结果摘要

### 2. 前台能看到线程态，不再只有聊天摘要

- 选择一条最近会话
- 预期：
  - 会话详情里出现线程状态卡片
  - 能看到投影待办数量、线程产物数量、subagent 数量
  - 能看到 workspace / uploads / outputs 的线程目录摘要
  - 如果任务复杂，能看到 subagent 列表而不只是步骤列表

### 3. 后台能看到 Deer runtime 关键对象

- 打开 `admin/access?tab=runtime`，选中同一任务
- 预期：
  - 出现 `Thread State / Deer Runtime`
  - 出现 `Subagent Runs`
  - `运行检查点` 中能看到 `workerIds / subagentRunIds`
  - 线程态里的 todo / artifact / latest checkpoint 能对应当前任务

### 4. 失败后可定位，而不是只给最终答复

- 选一条失败任务或故意制造一个失败请求
- 预期：
  - 前台或后台能明确看到失败步骤
  - 后台日志和 checkpoint 能看出失败位置
  - “整轮重试 / 失败步骤重试 / 恢复续跑”按钮可以点击

### 5. 页面交互具备基本产品质感

- 键盘 tab 到按钮和任务卡片
- 预期：
  - 有可见 focus 态
  - 文案状态不是一堆原始英文状态码
  - 卡片、按钮、输入区层级清楚，没有明显挤压或错位

## P1 建议通过

### 6. 多步任务能看出拆解

- 测试提示词建议：
  - “帮我查本周审批记录，再整理成汇报摘要”
  - “帮我分别汇总未读消息和审批待办，并给出下一步建议”
- 预期：
  - 至少出现多个步骤
  - 理想情况下能看到 subtask / worker / subagent

### 7. 同一线程有连续性

- 在同一线程连续追问两轮
- 预期：
  - 同一线程的目录、待办、产物持续更新
  - 后台 `lastTaskId / lastRunId` 会跟着变化

### 8. 页面在中等宽度下可读

- 把浏览器宽度缩到大约 1024px 或更低
- 预期：
  - 工作台卡片能换行
  - 运行观测面板不会横向炸开
  - 长路径会换行，不会把布局撑坏

## 建议测试提示词

### 任务型

- “帮我把今天校园事项整理成三段式工作简报”
- “请把我本周审批和消息拆成优先级清单”

### 多步型

- “先查审批记录，再根据记录整理成汇报文档”
- “分别汇总我本周未读消息和待审事项，再输出行动建议”

### 容错型

- “只重试上一轮失败步骤”
- “从刚才卡住的位置继续往下执行”

## 当前不足

### 1. subagent 还不是完整的多代理系统

目前的 subagent 更像“可追踪的并发执行单元”，已经比裸 `Promise.all` 强，但还不是 OpenClaw / deer-flow 那种真正带独立规划、独立上下文和更强自治能力的深层子代理。

### 2. 前台工作台还不是完整控制台

现在前台 `firefly-workbench` 已经能看到线程态和 subagent 摘要，但仍偏“任务会话总览页”，还不是完整的任务控制面。更细的 replay、checkpoint、事件排障仍主要在后台。

### 3. 对话主页面没有同步升级到同等 runtime 颗粒度

这轮刻意没有大改你原来的对话页，所以主对话页仍然相对克制。优点是没破坏你现有使用习惯，缺点是 runtime 能力更多体现在工作台和后台，不是直接体现在聊天界面本体。

### 4. 复杂拆解仍依赖 planner 命中

如果问题太简单，planner 可能不会拆出明显的 subtask / worker / subagent。也就是说，现在的“可观测性”已经具备，但“是否触发复杂结构”还受问题类型影响。

### 5. 产物沉淀还偏运行态，不够业务态

当前 artifact 已经写入线程态，但前台还没有做成“可打开、可预览、可下载、可复用”的业务化产物中心，这部分还需要继续做。

### 6. 还缺自动化验收

目前主要通过页面人工验收和构建验证，缺 Playwright 级别的关键链路 E2E 和样式回归测试。

## 如果明天验收不过，优先继续改这三项

1. 把前台 workbench 的线程详情进一步做成“任务控制台”，补 checkpoint 和 artifact 预览
2. 把主对话页再收一层 runtime 暴露信息，让用户不离开聊天页也能感知任务状态
3. 给复杂任务补更稳定的拆解策略，降低“有时能拆、有时拆不出来”的波动
